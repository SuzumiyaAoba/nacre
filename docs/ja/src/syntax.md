# 言語設計

Nacre は、ソースコードとは別に権限をレビューできる型付きスクリプトを目指して
設計されています。

## 設計目標

1. 一般的なスクリプトのデータフローを簡潔に記述し、静的に検査する
2. 実行時にコンパイラを必要としない、単体の Bash へコンパイルする
3. 実行ファイルとファイルシステムの権限をソース管理下のコードから分離する
4. シェルコマンド文字列を組み立てず、引数の境界を保持する
5. 安全でない構文を無害化しようとせず、明示的に拒否する

## 権限モデル

```text
Nacre ソース
    │ 構文解析・型検査
    ▼
静的ケイパビリティ名 ────── 外部 TOML ポリシー
    │                               │
    └──────── 権限を解決 ──────────┘
                    │
                    ▼
              単体の Bash
```

ソースは静的なケイパビリティを要求できます。

```nacre
const text = run.read.document("input/data.txt")
run.output.print(text)
```

その名前を実行ファイルへ対応付けられるのはポリシーだけです。

```toml
[filesystem]
read = ["input"]

[command_groups.read.commands.document]
program = "bin/read-document"
args = 1
read_args = [0]

[command_groups.output.commands.print]
program = "bin/print"
args = 1
```

## 安全プロファイル

安全プロファイルでは、任意のシェル文字列、生の Bash、動的な実行ファイル名、
パイプライン、リダイレクト、バックグラウンドコマンドを提供しません。複雑な操作は、
用途を限定してレビューしたプログラムへ実装し、ポリシーから名前を指定します。

パーサーの内部表現には歴史的な構文の一部が残っていますが、公開コンパイル処理では
出力前に拒否されます。ドキュメントが説明するのは内部形式ではなく、受理される
安全プロファイルです。

## ランタイム境界

生成された Bash は、必要に応じて構造化された値、クロージャ、パスガード用の固定
ヘルパーを含みます。これらはコンパイラが所有するランタイムコードであり、Nacre の
ソースから選択または置換できません。

ポリシー、許可された実行ファイル、コンパイラ、ランタイムヘルパーは信頼されます。
ソースから与えられた値は、コマンド境界を越えてもデータとして保持されます。

## 実装上の概要

- 構文解析には宣言的な `rust-peg` 文法を使用
- Bash 出力の前に型検査と lowering を実行
- モジュールの import を展開し、名前空間を付けてから検査
- 生成されたスクリプトで `set -euo pipefail` を有効化

厳密な現在の動作は、この概要ではなく実装とテストによって定義されます。
