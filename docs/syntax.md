# Nacre Safe Language Design

Nacre is designed for code generated by Coding Agents. Its execution model
separates source code from authority.

## Authority Model

1. Nacre source describes typed computation and static capability calls.
2. A separately reviewed TOML policy grants command and filesystem authority.
3. The compiler resolves command aliases to canonical files.
4. Generated Bash passes arguments individually without shell evaluation.
5. Runtime path guards enforce canonical read and write roots.

Source code cannot declare or broaden its own policy.

## Command Syntax

```nacre
const text = run.read.cat("input/data.txt")
run.output.print(text)
```

The grammar accepts only `run.<group>.<command>(arguments...)`. Group and
command names are identifiers known at compile time. A policy can define
read-only groups by assigning `read_args` and omitting all write roots and
write-capable commands.

## Policy Syntax

```toml
[filesystem]
read = ["input"]
write = ["output"]

[command_groups.read.commands.cat]
program = "/bin/cat"
read_args = [0]

[command_groups.output.commands.print]
program = "scripts/print"
```

Unknown policy fields, invalid identifiers, missing programs, overlapping
read/write argument indexes, and invalid roots are compile errors.

## Removed Capabilities

The language does not expose arbitrary shell strings, raw Bash, dynamic
executable names, shell pipelines, shell redirects, or dynamic `bash -c`
execution. Trusted scripts may implement a complex operation, but the script
itself must be explicitly named by the external policy.

## Security Boundary

The policy and every approved executable or script are trusted inputs. Nacre
protects against source-controlled executable selection, argument splitting,
shell interpolation, and paths outside configured roots. Like other
path-checking systems implemented in shell, it cannot eliminate filesystem
time-of-check/time-of-use races against a concurrently malicious process.

All code in this repository was developed by a Coding Agent.
